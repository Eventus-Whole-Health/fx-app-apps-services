    **Backend proxy routes** — Create `keystone-platform/backend/app/routes/scheduler/` module that proxies requests to the fx-app-apps-services scheduler API. Follow the pattern from `routes/monitoring/` but instead of direct SQL queries, forward HTTP requests to the fx-app-apps-services endpoint.

    In `proxy.py`:
    - Use `httpx.AsyncClient` to forward requests to `fx-app-apps-services` base URL
    - Read the base URL from env var `SCHEDULER_API_URL` (default: `https://fx-app-apps-services.azurewebsites.net`)
    - Proxy these routes:
      - `GET /services` → `GET /api/scheduler/services`
      - `GET /health` → `GET /api/scheduler/health`
      - `GET /services/{service_id}/history` → `GET /api/scheduler/services/{service_id}/history` (pass through query params: page, page_size, status, start_date, end_date)
      - `POST /services` → `POST /api/scheduler/services` (pass through JSON body)
      - `PUT /services/{service_id}` → `PUT /api/scheduler/services/{service_id}` (pass through JSON body)
      - `DELETE /services/{service_id}` → `DELETE /api/scheduler/services/{service_id}`
      - `POST /services/{service_id}/trigger` → `POST /api/scheduler/services/{service_id}/trigger`
    - All routes use `monitoring_roles = require_any_role("apps.admin", "Admin")` for RBAC (same as monitoring module)
    - Return the upstream JSON response directly with matching status codes
    - Handle connection errors gracefully (return 502 if upstream unreachable)

    In `__init__.py`:
    - Create `router = APIRouter()` and include `proxy.router` with no prefix (prefix set at main.py level)

    In `main.py`:
    - Add `from app.routes.scheduler import router as scheduler_router`
    - Add `app.include_router(scheduler_router, prefix="/api/scheduler", tags=["scheduler"])` alongside existing monitoring router

    **Frontend types** — Create `keystone-platform/frontend/src/types/scheduler.ts`:
    ```typescript
    // Matches the response shape from GET /api/scheduler/services
    export interface SchedulerService {
      id: number
      function_app: string
      service: string
      trigger_url: string
      frequency: string
      schedule_config: string | null
      is_active: boolean
      status: string
      last_triggered_at: string | null
      next_run_time: string | null
      health_status: 'healthy' | 'degraded' | 'failing'
      failure_count: number
      total_recent: number
      retry_count: number
      max_retries: number
      log_id: number | null
      error_message: string | null
    }

    export interface HealthSummary {
      total: number
      healthy: number
      degraded: number
      failing: number
    }

    export interface ExecutionHistoryEntry {
      log_id: number
      function_app: string
      service_name: string
      started_at: string
      ended_at: string | null
      duration_ms: number | null
      status: string
      error_message: string | null
      request_data: string | null
      response_data: string | null
    }

    export interface HistoryResponse {
      data: ExecutionHistoryEntry[]
      page: number
      page_size: number
      total_pages: number
      total_count: number
    }

    // Grouped by function_app for card grid
    export interface FunctionAppGroup {
      function_app: string
      services: SchedulerService[]
      health_status: 'healthy' | 'degraded' | 'failing'
      total_services: number
      error_count: number
    }
    ```

    **Frontend API service** — Create `keystone-platform/frontend/src/services/scheduler-api.ts`:
    - Follow the pattern from `monitoring-api.ts` (SWR hooks, same fetcher pattern, API_URL from env)
    - Export hooks:
      - `useSchedulerServices()` — calls `GET /api/scheduler/services`, returns services array + groups them by function_app into `FunctionAppGroup[]` using `useMemo`
      - `useSchedulerHealth()` — calls `GET /api/scheduler/health`, returns `HealthSummary`
      - `useServiceHistory(serviceId: number | null, page?: number, pageSize?: number)` — calls `GET /api/scheduler/services/{id}/history`, conditionally fetches only when serviceId is provided
    - Export mutation functions (not hooks — called imperatively):
      - `createSchedule(data)` — POST to `/api/scheduler/services`
      - `updateSchedule(id, data)` — PUT to `/api/scheduler/services/{id}`
      - `deleteSchedule(id)` — DELETE to `/api/scheduler/services/{id}`
      - `triggerService(id)` — POST to `/api/scheduler/services/{id}/trigger`
    - Use `refreshInterval` parameter on `useSchedulerServices` for auto-refresh (45 seconds — per Claude's discretion within 30-60s range)
    - The fetcher should use `credentials: 'include'` like monitoring-api.ts
    - For grouping in `useSchedulerServices`: sort function_app names alphabetically per user decision, compute aggregate health (worst health_status among services), compute error_count (count of services where health_status !== 'healthy')
