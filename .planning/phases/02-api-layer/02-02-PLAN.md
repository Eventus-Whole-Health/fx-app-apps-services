Add three CRUD endpoints to the existing Blueprint in `scheduler_endpoints.py`.

**Validation helper function** — create `validate_schedule_input(data: Dict) -> Tuple[bool, Optional[str]]`:
- `frequency` must be one of: `once`, `daily`, `weekly`, `monthly`, `hourly` (case-insensitive, normalize to lowercase)
- `trigger_url` must be non-empty string
- `function_app` must be non-empty string
- `service` (service name) must be non-empty string
- `schedule_config` if provided must be valid JSON (try `json.loads()`)
- Return `(True, None)` if valid, `(False, "error message")` if invalid

**1. POST `/api/scheduler/services` (Create)**

Accept JSON body:
```json
{
  "function_app": "ai-scribing-services",
  "service": "new-daily-sync",
  "trigger_url": "https://fx-app-ai-scribing-services.azurewebsites.net/api/daily-sync",
  "frequency": "daily",
  "schedule_config": "{\"start_time\": \"06:00\", \"end_time\": \"22:00\"}",
  "json_body": "{}",
  "start_date": "2026-03-01T06:00:00",
  "trigger_limit": null,
  "max_retries": 3,
  "max_execution_minutes": 30
}
```

Implementation:
1. Parse request body, validate with `validate_schedule_input()`
2. If invalid, return 400: `{"error": "Validation failed", "detail": "..."}`
3. Build INSERT query:
```sql
INSERT INTO jgilpatrick.apps_central_scheduling
    (function_app, service, trigger_url, frequency, schedule_config,
     json_body, start_date, trigger_limit, max_retries, max_execution_minutes,
     is_active, status, triggered_count, retry_count)
VALUES
    ('{function_app}', '{service}', '{trigger_url}', '{frequency}', '{schedule_config}',
     '{json_body}', '{start_date}', {trigger_limit}, {max_retries}, {max_execution_minutes},
     1, 'pending', 0, 0)
```
4. Use `method="execute"` since this is an INSERT.
5. After insert, query back the newly created row by matching function_app + service (since we cannot use SCOPE_IDENTITY via SQL Executor API — this is a known limitation from ARCHITECTURE.md).
6. Return 201: `{"success": true, "data": {"service": {...created_service...}}}`
7. Sanitize ALL user-provided string values with `sanitize_sql_string()` before SQL insertion.
8. Handle NULL values for optional fields (trigger_limit, schedule_config, json_body, max_execution_minutes) — use SQL NULL, not the string 'None'.

**2. PUT `/api/scheduler/services/{service_id}` (Update)**

Accept JSON body with any subset of updatable fields:
```json
{
  "frequency": "weekly",
  "schedule_config": "{\"day_of_week\": \"monday\", \"start_time\": \"06:00\"}",
  "trigger_url": "https://...",
  "is_active": true
}
```

Implementation:
1. Look up existing service by ID. If not found, return 404.
2. Parse request body. If `frequency`, `trigger_url`, or `function_app` are provided, validate them.
3. Build dynamic UPDATE query — only SET columns that were provided in the request body.
4. Updatable fields: `function_app`, `service`, `trigger_url`, `frequency`, `schedule_config`, `json_body`, `start_date`, `trigger_limit`, `max_retries`, `max_execution_minutes`, `is_active`.
5. Do NOT allow updating: `id`, `status`, `triggered_count`, `last_triggered_at`, `retry_count`, `next_retry_at`, `processed_at`, `log_id` (these are system-managed).
6. After update, query back the updated row and return: `{"success": true, "data": {"service": {...updated_service...}}}`
7. Sanitize all string values. Handle boolean `is_active` (convert Python bool to SQL bit: 1 or 0).

**3. DELETE `/api/scheduler/services/{service_id}` (Soft Delete)**

Implementation:
1. Look up service by ID. If not found, return 404.
2. Soft delete: `UPDATE jgilpatrick.apps_central_scheduling SET is_active = 0 WHERE id = {service_id}`
3. Return 200: `{"success": true, "data": {"id": service_id, "deleted": true}}`
4. Per CONTEXT.md: this is a soft-delete (set is_active=0), NOT a hard delete.

**Common patterns for all CRUD endpoints:**
- Use `async with SQLClient() as sql:` pattern
- Wrap in try/except with 500 error response
- Use `auth_level=func.AuthLevel.ANONYMOUS`
- Log operations with LOGGER
- All string inputs sanitized before SQL insertion

**1. Add manual trigger endpoint to `scheduler_endpoints.py` (API-04):**

**POST `/api/scheduler/services/{service_id}/trigger`**

Per CONTEXT.md, this is fire-and-forget: endpoint returns immediately after marking the service as processing. The dashboard polls `/api/status/{log_id}` for completion.

Implementation:
1. Parse `service_id` from route params. Validate it is numeric.
2. Look up service from `apps_central_scheduling` by ID. If not found or `is_active = 0`, return 404.
3. Import and call `process_scheduled_services_with_overrides` from the scheduler module:
   ```python
   from functions.scheduler.timer_function import process_scheduled_services_with_overrides
   ```
4. Call it with `force_service_ids=[service_id]` and `bypass_window_check=True`.
5. This function already handles:
   - Marking the service as processing
   - Triggering the HTTP request
   - Logging to `apps_master_services_log`
   - Handling 202 polling
   - Updating status on completion
6. The function returns a results dict. Extract the `log_id` from the results if available.
7. Return 200 immediately:
```json
{
  "success": true,
  "data": {
    "service_id": 1,
    "triggered": true,
    "message": "Service triggered successfully",
    "log_id": 5001,
    "status_url": "/api/status/5001"
  }
}
```

**Important consideration:** `process_scheduled_services_with_overrides` is an async function that processes the service synchronously (it awaits the full execution including polling). For fire-and-forget behavior per CONTEXT.md, we need to either:

**Option A (Simpler):** Call it and await the result. The endpoint will block until the service completes. This is acceptable because the Keystone ASP has unlimited timeout and the dashboard can handle long-running requests. However, this contradicts "fire-and-forget" from CONTEXT.md.

**Option B (Preferred per CONTEXT.md):** Use `asyncio.create_task()` to run the scheduler function in the background, and return immediately with a log_id. But we need the log_id BEFORE the function completes. Approach:
1. First, manually create a log entry using ServiceLogger.log_start() to get a log_id
2. Then start the background task
3. Return the log_id immediately

However, `process_scheduled_services_with_overrides` creates its own log entries internally. To avoid double-logging, use a simpler approach:

**Option C (Best fit):** Since the existing manual trigger endpoint (`POST /api/scheduler/manual-trigger`) already handles this exact flow and the dashboard can poll `/api/status/{log_id}`, implement the new endpoint as a thin wrapper that:
1. Validates the service exists and is active
2. Calls `process_scheduled_services_with_overrides(force_service_ids=[int(service_id)], bypass_window_check=True)`
3. Extracts any log_ids from the returned results
4. Returns the result with log_id(s) for polling

The existing function already logs to `apps_master_services_log`, so the dashboard gets a trackable execution.

Go with Option C. The endpoint awaits the full execution (unlimited timeout handles this). The response includes log_id(s) and status_url(s) for the dashboard to show results.

If the result dict from `process_scheduled_services_with_overrides` doesn't directly expose log_ids, query `apps_master_services_log` for the most recent log entry matching this service after the trigger completes:
```sql
SELECT TOP 1 log_id
FROM jgilpatrick.apps_master_services_log
WHERE function_app = '{function_app}'
AND service_name = '{service_name}'
ORDER BY started_at DESC
```
