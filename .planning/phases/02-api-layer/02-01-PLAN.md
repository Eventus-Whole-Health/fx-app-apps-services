**1. GET `/api/scheduler/services` (API-01: List schedules with health)**

Query `jgilpatrick.apps_central_scheduling` for all services where `is_active = 1`. For each service, compute:

- **Health status** by querying last 5 rows from `jgilpatrick.apps_master_services_log` where `function_app` matches and `service_name` matches (use the `service` column from scheduling table as `service_name`). Health logic:
  - If no log entries exist: `"healthy"` (never failed)
  - If 0 failures in last 5: `"healthy"`
  - If 1-2 failures in last 5: `"degraded"`
  - If 3+ failures in last 5: `"failing"`
  - If current scheduling status is `'processing'` and `processed_at` older than `COALESCE(max_execution_minutes, 30)` minutes: `"failing"` (stuck)

- **Next run time** computed in Python from `frequency`, `schedule_config`, and `last_triggered_at`:
  - `hourly`: last_triggered_at + 1 hour (or next hour mark if no last_triggered_at)
  - `daily`: Parse `schedule_config` for time window, return next occurrence
  - `weekly`: Parse day_of_week from config, return next matching day
  - `monthly`: Parse day_of_month from config, return next matching day
  - `once`: NULL (already triggered or waiting for window)
  - If computation fails or is complex, return `null` — dashboard handles gracefully

- **Response shape** (consistent envelope per CONTEXT.md):
```json
{
  "success": true,
  "data": {
    "services": [
      {
        "id": 1,
        "function_app": "ai-scribing-services",
        "service": "daily-transcription-sync",
        "trigger_url": "https://...",
        "frequency": "daily",
        "schedule_config": "{...}",
        "is_active": true,
        "status": "pending",
        "last_triggered_at": "2026-02-27T10:00:00",
        "next_run_time": "2026-02-28T10:00:00",
        "health_status": "healthy",
        "retry_count": 0,
        "max_retries": 3,
        "next_retry_at": null,
        "triggered_count": 142
      }
    ],
    "count": 43
  }
}
```

Use a single SQL query with a subquery or CTE to get health data efficiently rather than N+1 queries. Approach: Use a CTE that ranks recent log entries per service, then LEFT JOIN to scheduling table. Example pattern:

```sql
WITH recent_logs AS (
    SELECT
        function_app,
        service_name,
        status,
        ROW_NUMBER() OVER (PARTITION BY function_app, service_name ORDER BY started_at DESC) as rn
    FROM jgilpatrick.apps_master_services_log
    WHERE started_at >= DATEADD(day, -7, GETDATE())
),
health_summary AS (
    SELECT
        function_app,
        service_name,
        COUNT(*) as total_recent,
        SUM(CASE WHEN status IN ('failed', 'error', 'timeout') THEN 1 ELSE 0 END) as failure_count
    FROM recent_logs
    WHERE rn <= 5
    GROUP BY function_app, service_name
)
SELECT
    s.*,
    COALESCE(h.failure_count, 0) as failure_count,
    COALESCE(h.total_recent, 0) as total_recent
FROM jgilpatrick.apps_central_scheduling s
LEFT JOIN health_summary h
    ON s.function_app = h.function_app
    AND s.service = h.service_name
WHERE s.is_active = 1
ORDER BY s.function_app, s.service
```

Then compute `health_status` in Python from `failure_count` and `total_recent`, plus the stuck-service check.

Compute `next_run_time` in Python after fetching results. Use a helper function `compute_next_run_time(service: Dict) -> Optional[str]` that handles each frequency type. Return ISO 8601 strings. If schedule_config is present, parse it as JSON for time windows and day specifications.

3. Build query against `jgilpatrick.apps_master_services_log`:

```sql
SELECT

    ended_at,
    duration_ms,
    error_message,
    request,
    response,
    trigger_source
FROM jgilpatrick.apps_master_services_log
WHERE function_app = '{function_app}'
AND service_name = '{service_name}'
{status_filter}
{date_filters}
ORDER BY started_at DESC
OFFSET {offset} ROWS
FETCH NEXT {page_size} ROWS ONLY
```
